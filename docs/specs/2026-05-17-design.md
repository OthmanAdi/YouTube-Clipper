**Status:** Draft awaiting user review

## 1. Problem

When Ahmad-Othman hears a great moment in a YouTube video, he loses 5+ minutes scrubbing, transcribing, and pasting it into class material. He needs a sub-30-second pipeline that produces a permanent, verbatim, AI-summarized markdown note from a Ctrl+drag on the YouTube seekbar.

## 2. Solution at a Glance

A Chrome MV3 extension paints a pink range on YouTube's seekbar when the user Ctrl+drags. On release, the extension calls a local FastAPI daemon at `127.0.0.1:7777`. The daemon runs a six-stage pipeline (resolve → download → normalize → transcribe → summarize → write_note) and pushes live progress back to the extension popup over WebSocket. Output is one folder per clip in `output/`, containing `note.md` (Obsidian-friendly), `audio.mp3`, `transcript.json`, `transcript.txt`, and `raw.log`.

Of the six stages, **only stage 5 (summarize) uses an LLM**. The other five are deterministic CLI-tool wrappers. Robustness comes from minimizing the AI surface.

## 3. Locked Decisions

## 5. Components

### 5.1 Chrome Extension (`apps/extension/`)

- **manifest.json** v3, requests `activeTab`, `storage`, host permission for `*://*.youtube.com/*`.
- **content script** (`src/content/`) injected into YouTube watch pages:
  - hooks `mousedown`, `mousemove`, `mouseup` on `.ytp-progress-bar`,
  - only acts when `event.ctrlKey === true`,
  - paints a `<div>` overlay positioned absolutely over the existing progress bar,
  - computes `start_s` and `end_s` from `(event.offsetX / barWidth) * video.duration`,
  - on release: sends `{url, start_s, end_s, video_title, channel_name}` to the background service worker.
- **background service worker** (`src/background/`):
  - `fetch('http://127.0.0.1:7777/clip', {method:'POST', body: …})` to enqueue,
  - opens a WebSocket to `/events/{job_id}`,
  - relays progress to the popup via `chrome.runtime.sendMessage`.
- **popup** (`src/popup/`):
  - plain HTML/CSS (no React — the popup is <300 lines),
  - shows source metadata, range, summarizer dropdown, Extract/Cancel,
  - while running: stage progress bar + last log line,
  - on done: `Open note ▸ Open folder ▸`,
  - on fail: `Retry from stage N ▸` + error class + first 3 log lines.
- **health chip:** the popup polls `GET /health` once on open; green dot when daemon is up, red dot with "Daemon not running. Run `scripts/start-daemon.ps1`" when down.

### 5.2 FastAPI Daemon (`apps/daemon/`)

- **HTTP routes:**
  - `POST /clip` — body `{url, start_s, end_s, summarizer}` → returns `{job_id, clip_id}`; queues job, returns immediately.
  - `GET /clips` — paginated index of all completed clips (reads `output/` manifests).
  - `GET /health` — daemon status, deps probe (yt-dlp, ffmpeg, whisper model loaded, Azure reachable, Ollama reachable).
  - `GET /jobs/{job_id}` — current state of a job (for popup reopen).
- **WebSocket route:**
  - `WS /events/{job_id}` — pushes stage transitions, per-stage progress percent, and the most recent log line for that job.
- **Worker:** single async worker, one concurrent job (avoids GPU/Whisper contention). A simple `asyncio.Queue` is enough — no Celery, no Redis.
- **Lifecycle:** uvicorn started by `scripts/start-daemon.ps1`. SIGINT triggers a graceful shutdown that finishes the current job and rejects new ones.

### 5.3 Pipeline Stages (`apps/daemon/youtube_clipper/pipeline/`)

Six modules, one per stage. Each exports a coroutine `async def run(job: Job, ctx: PipelineContext) -> Job` that:
1. Reads required inputs from `job.manifest` and disk.
2. Performs work, emitting structured logs tagged with `stage=...`.
3. Writes outputs to the clip folder.
4. Updates `job.manifest` (atomic write of `manifest.json`).
5. Returns the updated job.

Each stage is idempotent. Re-running with existing outputs skips work unless explicitly forced.

| Stage | Type | Tool | Outputs |
|-------|------|------|---------|
| 1 resolve   | det. | `yt-dlp -J` | `metadata.json` |
| 2 download  | det. | `yt-dlp --download-sections "*Ss-Es" -x` | `audio.raw.<ext>` |
| 3 normalize | det. | `ffmpeg` | `audio.mp3`, `audio.meta.json` |
| 4 transcribe | det. | `faster-whisper large-v3` | `transcript.json`, `transcript.txt` |
| 5 summarize | **AI** | Azure Foundry **or** Ollama via `httpx` | `summary.json` |
| 6 write_note | det. | Jinja2 | `note.md`, updates `manifest.json` |

### 5.4 Summarizer Adapters (`apps/daemon/youtube_clipper/adapters/`)

A common interface:

```python
class Summarizer(Protocol):
    name: str  # "azure-foundry/gpt-4o-mini" or "ollama/qwen2.5:14b"
    async def summarize(self, transcript: str, *, language: str) -> SummaryResult: ...

class SummaryResult(BaseModel):
    tldr: str          # 40-80 words
    bullets: list[str] # 3-7 items
    tags: list[str]    # 0-5 short kebab-case tags
    raw_response: dict # for raw.log
```

Two implementations: `AzureFoundryAdapter`, `OllamaAdapter`. Both speak HTTPx, both request **structured JSON output** (Azure: response_format json_object; Ollama: prompt + JSON-mode flag). Both validate with `SummaryResult.model_validate_json` — on parse failure, one retry with a stricter system prompt, then surface to user.

### 5.5 Logging (`apps/daemon/youtube_clipper/logging.py`)

`structlog` configured with:
- JSON renderer (jsonlogger),
- timestamper (ISO 8601 with +02:00 offset),
- contextvars merging (so `job_id` / `clip_id` / `stage` are auto-attached once set at job start),
- dual output: stderr (for `start-daemon.ps1` terminal) and `logs/pipeline-YYYY-MM-DD.jsonl` (rotated daily via `logging.handlers.TimedRotatingFileHandler`).

Each clip's `raw.log` is built post-job by filtering `logs/pipeline-*.jsonl` for that `job_id` and writing the matching lines into the clip folder.

## 6. Data Model

```python
class Job(BaseModel):
    job_id: str           # j_2026-05-17_001
    clip_id: str          # 2026-05-17-001
    created_at: datetime
    state: JobState       # pending|queued|running|done|failed
    current_stage: Stage | None
    stages_done: list[Stage]
    failed_at_stage: Stage | None
    error_class: str | None
    error_message: str | None
    retry_count: dict[Stage, int]
    input: ClipInput
    paths: ClipPaths      # absolute paths to job_dir, audio, transcript, note
    durations_ms: dict[Stage, int]
    summarizer_used: str | None
```

`manifest.json` in each clip folder is `Job.model_dump_json(indent=2)`. It is the source of truth for that clip — read on re-run, on `Retry from stage`, on `/jobs/{id}` lookup.

### 7.1 "My Notes" Preservation Rule

`## My Notes` is **always the last section in the template.** This makes preservation trivial: everything from the line `## My Notes` to end-of-file is the user's, and the writer never touches it.

On re-render (e.g. user clicked `Retry from stage 5` after editing notes manually), the writer:
1. Reads the existing `note.md` if present.
2. Extracts everything from the first occurrence of `^## My Notes$` to EOF (inclusive of the heading).
3. Renders the new note from the Jinja template *up to but not including* `## My Notes`.
4. Concatenates: `{rendered_top}\n{preserved_tail}`.
5. Writes atomically (`tmp` + rename).

If `note.md` does not exist, the section is created from the template with the placeholder comment.

Pipeline run details (model, durations, summarizer) live in the YAML frontmatter only — no body section to drift between renders.

### 7.2 Tag Flow

The summarizer adapter returns `tags: list[str]` as part of `SummaryResult`. Stage 6 writes those values into the `tags:` field of the YAML frontmatter. Frontmatter `tags:` is the single source of truth — Obsidian and qmd both index from there. The body has no `## Tags` section.

## 8. Range Selection UX

**Modifier:** `Ctrl` only. Plain click/drag remains normal seek behavior.

**On Ctrl+mousedown on `.ytp-progress-bar`:**
- start a drag session,
- record `start_x` and freeze YouTube's hover preview thumb,
- pause the video (optional, configurable),
- show a tooltip at the cursor: `0:23`.

**On Ctrl+mousemove during drag:**
- update the overlay's left/width to span from `start_x` to current `x`,
- update tooltip to show `start -> end (length)`.

**On Ctrl+mouseup:**
- finalize `start_s` and `end_s` (sorted ascending — backwards drag normalizes),
- if length < 2s: discard with a toast "Range too short, ignored.",
- otherwise: open the extension popup with the range pre-filled.

**Edge cases:**
- Ctrl released mid-drag: drag is cancelled, overlay removed.
- Window blur during drag: drag cancelled.
- Live stream: stage 2 will reject; UX shows a clear "Live streams not yet supported" pre-check in the popup based on `metadata.is_live` from a prefetch.

## 9. Error Handling and Retry

**Per-stage policy:**

| Stage | Retry on | Max attempts | Backoff | Fallback |
|-------|----------|--------------|---------|----------|
| 1 resolve | network errors | 3 | 2s, 4s, 8s | none |
| 2 download | network, throttling | 4 | 5s, 15s, 30s, 60s | force `--format bestaudio/best` |
| 3 normalize | none | 1 | — | none |
| 4 transcribe | CUDA OOM | 3 | immediate | fp16 → int8 → CPU |
| 5 summarize | 429, 5xx, timeout, JSON-parse | 3 | 1s, 4s, 10s | user prompt: switch backend? |
| 6 write_note | filesystem busy | 3 | 1s, 1s, 1s | none |

**Failure is not deletion.** Failed jobs keep `output/<clip_folder>/` intact for inspection. `scripts/doctor.ps1 --prune-failed` cleans them when explicitly requested.

**`Retry from stage N`** in the popup posts `POST /clip/{job_id}/retry?from=N`, which:
1. deletes outputs from stage N onward in the clip folder,
2. resets `retry_count[N..]` to zero,
3. re-queues the job at stage N.

## 10. Logging Architecture

**Stack:** `structlog` -> JSONL -> rotating file (`logs/pipeline-YYYY-MM-DD.jsonl`) + stderr.

**Every log line includes:**
- `ts` — ISO 8601 with offset
- `level` — debug | info | warning | error | critical
- `job_id`, `clip_id` (from contextvars)
- `stage`
- `event` — short dotted name, e.g. `download.bytes_received`, `whisper.completed`
- domain fields (per event)

**Levels:**
- `debug` — exact subprocess command lines, full stdout/stderr
- `info` — stage starts/ends, durations, counts, model info
- `warning` — retries, fallbacks, slow stages (> p95 of prior runs)
- `error` — stage failures with stack
- `critical` — daemon-level: port collision, GPU lost, model failed to load

**Per-clip `raw.log`** is a derived file: post-job, the daemon greps `logs/pipeline-*.jsonl` for the job's `job_id` and writes the matching lines (chronological) to the clip folder. Easy human inspection without grep skill.

**Extension popup** receives the last 5 `info` lines for the current job over WS, displaying the latest one as "Last log: ...".

## 13. Testing Strategy

**Three rings:**

1. **Unit** (~50 tests, fast, no network): each stage as a pure function with mocked subprocesses; URL parsing, time parsing, retry logic, slug generation, manifest read/write, My-Notes preservation, summary JSON validation.
2. **Integration** (~10 tests, real subprocesses, real Whisper, network-required): three short Creative-Commons-licensed YouTube IDs committed by reference. Full pipeline on each. Skipped automatically when offline.
3. **E2E** (~3 tests with Playwright + real Chrome + extension loaded): spin up daemon on test port, navigate to a YouTube fixture, simulate Ctrl+drag, assert `note.md` exists with expected sections.

**LLM testing**: `RecordReplay` mode for adapters. First run hits the real Azure/Ollama and captures responses to `tests/fixtures/llm/<hash>.json`. Subsequent test runs replay from disk — zero token cost, deterministic.

## 14. Bootstrap (`scripts/`)

- `install.ps1` — installs Python deps via uv, installs ffmpeg + yt-dlp to `C:\Tools\` if missing, builds extension, prints "Load unpacked at: …" path.
- `start-daemon.ps1` — `uv run uvicorn youtube_clipper.api.app:app --host 127.0.0.1 --port 7777 --log-level warning` (structlog handles real logging).
- `stop-daemon.ps1` — finds and SIGINTs the uvicorn process.
- `doctor.ps1` — probes: yt-dlp version, ffmpeg version, CUDA available, Whisper model present (downloads if missing), Azure endpoint reachable, Ollama endpoint reachable + model pulled, output dir writable, port 7777 free. Prints a green/red table. Exit 0 only if everything is green.

## 15. Out of Scope (explicit YAGNI)

- Multi-user / cloud sync.
- Authentication (daemon binds 127.0.0.1, never exposed externally).
- Video output. Audio only.
- Firefox / Edge / Safari ports.
- Mobile.
- A built-in clip browser UI. `note.md` in Obsidian/VS Code IS the UI.
- Auto-push to qmd / OpenMark. Defer to a post-v1 integration spec.
- Live streams.
- Auto-translate the summary into a different language than the transcript.
- Speaker diarization.

## 17. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| YouTube changes seekbar DOM | medium | high | content script uses player API (`document.querySelector('video').duration`) for time math, not DOM measurements alone. Selector kept in one constant. |
| yt-dlp range-download regression | low | high | pin yt-dlp version; `doctor.ps1` reports the version; integration tests catch this on the 3 fixtures. |
| CUDA OOM on long clips | medium | medium | stage 4 retry policy: fp16 -> int8 -> CPU. `max_range_seconds` config caps user input. |
| Azure cost runaway | low | medium | rate-limit per minute via async-semaphore; transcript-only summary (no audio upload); fixed model `gpt-4o-mini` not 4o. |
| Ollama model not pulled | medium | low | `doctor.ps1` detects and prompts the exact `ollama pull qwen2.5:14b` command. |
| Daemon port collision | low | medium | startup validation; clear error with suggestion to change `config.toml` port. |
| Disk fills with clips | low | medium | output folders are small (audio is mp3 not wav); `doctor.ps1 --usage` reports total size. |

## 19. Definition of Done for v1

1. `doctor.ps1` reports all green on a fresh user machine after `install.ps1`.
2. Ctrl+drag on any non-live YouTube video produces a complete `note.md` in under 30 seconds for a 2-minute clip on the user's hardware.
3. Killing the daemon mid-job and rerunning resumes from the right stage with no data loss.
4. `Retry from stage N` from the popup works for every stage.
5. Editing `## My Notes` and re-running the pipeline preserves the user's edits exactly.
6. Every clip folder has a complete `raw.log` correlated by `job_id`.
7. Switching the summarizer dropdown between Azure and Ollama produces equivalent-shape `summary.json` (same fields, same validation).
